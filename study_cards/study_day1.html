过关标准 聊 10 轮不丢上下文——第 10 轮问"我第一句说了什么"为什么能答对？
  • 大模型本身是 无状态、无记忆 的：它不会自己记住上一轮说过什么，每次请求对它都像"第一次见面"。
  • 真正的"记忆"在 你的程序里那个 messages 列表——每轮把 user、assistant 消息 append 进去，从不丢弃。
  • 每次调 API 都把 整段 messages 重新发过去。所以第一句 user 消息到第 10 轮仍然在列表里、仍然被发送，模型自然"看得到"，就能答对。
  • 只有 /clear（重置成只剩 system），或历史超出上下文窗口被 截断 时，才会真的"忘"。
messages = [
  {role:"system",    ...},
  {role:"user",      "北京天气?"}   ← 第 1 句，一直都在
  {role:"assistant", "晴，20°"},
  ...  （中间 8 轮）
  {role:"user",      "我第一句说了啥?"}  ← 第 10 轮，待回答
]  # 整个列表一起发 → 第 1 句还在里面
口诀不是模型有记忆，是你每次把完整历史重新喂给它。
02
过关标准 口头解释 system / user / assistant / tool 四种 role 各自的作用
system
最高优先级的"人设 / 规则"。设定助手的身份、语气、边界。一般放在开头、只放一次。
user
用户的输入——提问、指令。你在代码里 append 的那句就是它。
assistant
模型自己的回复。多轮里必须把它 append 回历史，否则下一轮模型"忘了自己刚说过啥"，对话会断。
tool
工具调用的返回结果（旧称 function）。流程：模型决定调某工具 → 你的代码执行 → 把结果以 role=tool 塞回 messages → 模型据此继续答。用于接实时数据、算数、外部 API。
口诀system 定规矩、user 提问、assistant 作答、tool 送外部结果——四色四职，交替成对话。
03
过关标准 temperature、max_tokens 是什么？temperature=0 与 1.5 各适合什么场景？
  • temperature 采样温度，控制输出的 随机性 / 发散度，常见范围 0~2。低=确定、保守、可复现；高=随机、有创意，也更容易跑偏。
  • max_tokens 单次回复的 输出长度上限（按 token 算，不含输入）。到达上限会被 截断——不是"回答完了"，是"写不下了"。
temperature = 0　冷静

要 稳定、可复现、事实准 的活：代码生成、数据抽取、分类、数学、每次答案都要一致的场景。

temperature = 1.5　发散

要 多样、有创意 的活：头脑风暴、写故事 / 取名、要多个不同角度的方案。（太高会语无伦次）

口诀temperature 管"多敢跳"，max_tokens 管"能写多长"——要准调 0，要野调高。
