01
循环骨架 工具调用是一场"来回"，不是一次问答
1. 用户提问一条 user 消息进历史。
2. 模型点单返回 tool_calls：要调谁、参数是啥。
3. 本地执行execute_tool 在你机器上真跑函数。
4. 结果回填role="tool" + tool_call_id 塞回历史。
5. 再问一遍带新历史重发，直到出文本才停。
  • 核心是一个循环：发请求 → 有 tool_calls 就执行并回填 → 再发请求，直到模型不再点工具、给出最终文本。
  • 一轮响应里可能有多个 tool_calls，必须 for 逐个执行、逐个回填，别漏。
  • 顺序铁律：先 append 带 tool_calls 的整条 assistant 消息，再 append 每个 tool 结果。顺序反了，模型对不上账，直接报错。
口诀用户问一次，模型和你的代码要来回好几趟；历史这本账，从头到尾是你的循环在记。
02
意图 vs 执行 模型真的执行函数了吗？——没有
模型 LLM
只吐一段 JSON：name + arguments，意思是"我想调 calculator"。这叫调用意图。它从头到尾一行代码都没跑。
你的代码
execute_tool 里 function(**args) 才是真正运行函数的地方。CPU 在你这边，执行发生在你的进程里。
所以
安全边界归你，因为跑代码的机器是你的。模型让算啥你就算啥，兜底得你来。
口诀模型只会"点单"，炒菜的是你的代码。这也是为什么工具的安全责任在你，不在模型。
03
模型怎么选工具 它唯一的依据，是你写的那句 description
  • 请求里 tools=tools 会把每个工具的说明书（name / description / parameters）一起发给模型——但不包含你的函数实现，模型看不到函数体。
  • 模型拿用户意图去和各工具的 description 做匹配，挑最像的那个，再自己填 arguments。
  • 所以 description 写烂 = 模型选错工具、填错参数。它不是笨，是你没把"什么时候该用我"讲清楚。
{
  "name": "calculator",
  // ↓ 模型选不选它，全看这一句
  "description": "执行数学表达式计算；需要数学运算时必须使用",
  "parameters": { /* expression: string */ }
}
口诀80% 的"模型不听话"，是 description 没写好，不是模型笨。工具描述要写清"什么时候用我"。
04
工具安全 calculator 用 ast 白名单，绝不用 eval
  • eval("字符串") 是把字符串当代码直接执行，给啥跑啥。喂它 __import__("os").system("del ...") 一样照删你的盘。
  • ast.parse 是只把字符串拆成语法树、不执行。拆开后你逐个节点检查，只放行认识的类型（数字、+−×÷）。
  • 恶意的 ast.Call（函数调用）节点不在白名单里，当场 raise，一行都没跑。这叫白名单：默认全拒，只放行明确允许的。
node = ast.parse(expr, mode="eval").body   # 只拆树，不执行

if isinstance(node, ast.Constant):          # 是数字 → 放行
    return node.value
elif isinstance(node, ast.BinOp):          # 是 +−×÷ → 放行
    ...
else:
    raise TypeError(...)                    # 其它一律拒绝
口诀eval 来者不拒，ast 先安检再放行。凡是处理不可信输入的工具，默认姿势就是"解析 + 白名单"。
05
循环兜底 while True 调 LLM 是要出事的：加 MAX_TURNS
  • while True = "信任模型会自己停"。万一模型抽风反复点工具，循环永不结束，每转一圈发一次 API，烧钱到爆账单。
  • 改成 for turn in range(MAX_TURNS)：最多转固定圈数，兜底刹车。真实产品没人敢用裸 while True 调模型。
  • for…else 的 else 只在"循环自然跑完、一次都没 break"时触发——正好对应"转满上限还没出答案"的异常情况。正常答完会 break，else 被跳过。
MAX_TURNS = 10

for turn in range(MAX_TURNS):
    message = send_messages(messages)
    messages.append(message)
    if not message.tool_calls:
        print(message.content)   # 先打印答案
        break                    # 再跳出 → else 被跳过
    for tool in message.tool_calls:
        ...                      # 执行工具、结果回填
else:
    print("[达到最大轮数，强制停止]")   # 只有转满才到这
口诀break 前先 print，else 挂外层 for；缩进错一格，else 挂到内层就全毁。while True 调模型 = 定时炸弹。
06
工程好习惯 让代码替你把意图说清楚（公司标准）
  • 魔法数字提常量：MAX_TURNS = 10 比循环里裸埋的 10 会说话。判据：这数字换个值会改变行为、或半年后看不懂它为啥是这个值 → 提；x + 1 里的 1 就别提，搞出 ONE = 1 反是灾难。
  • 用不到的循环变量写 _：它是给读者递信号"我故意不用、不是漏了"。或者干脆拿来打 第 N 轮 日志。最忌"定义了又不碰"的半吊子。
  • import 放模块顶部：别塞进函数体（除非可选依赖或启动很慢的重包才延迟导入）。—— 这是下次会当场纠的那类小习惯。
心法常量、_、import 位置背后是同一件事：代码要替你把意图说清楚，别让三个月后的自己去猜。这就是"能跑"和"能维护"的分界线。
