过关标准 为什么产品都用流式？（首 token 延迟 vs 总时长）
  • 总时长：发请求 → 最后一个 token 生成完。长回复十几秒，流式和非流式基本一样（流式甚至略多一点点开销）。
  • 首 token 延迟（TTFT）：发请求 → 第一个字冒出来，通常几百毫秒。
  • 非流式让用户干等「总时长」——盯着空白屏十几秒，体感像卡死；流式把等待压到 TTFT，字一出现就能读。流式优化的不是"快多少"，而是"感知延迟"。
  • 加成：人读字通常慢于模型生成，读着读着就生成完了，几乎无等待感；看到跑偏还能 early stop 省 token。

02
过关标准 上下文装不下时的 3 种策略？（截断 / 摘要压缩 / 检索）
截断
丢最早的对话轮，保留 system + 最近 N 轮。✔ 最简单（今天做的就是它，超 4000 token 丢最早）　✘ 会真忘掉早期信息
摘要压缩
把早期对话喂给模型压成一段摘要，用摘要替换原文。✔ 省 token 又留大意　✘ 丢细节 + 每次多一次 LLM 调用
检索 RAG
历史/文档进向量库，每轮只按当前问题检索最相关的几段塞进上下文。✔ 上下文再大也装得下　→ 第 2 周 RAG 的伏笔
口诀装不下三条路：砍掉（截断）、压扁（摘要）、外挂（检索）——越往后越保信息，也越费工。
03
实现要点 · 踩坑 stream=True 怎么写才对？resp 与 for 循环的数据流
  • 正文在 delta.content（不是非流式的 message.content）；想记 usage 必须加 stream_options={"include_usage":True}，且 usage 只在最后一个 choices 为空的 chunk 里。
  • for chunk in resp 是阻塞式：next() 嘬不到数据就原地等，消费端追不过生产端，不会打印一半就截断。resp 只是「插进服务器的吸管」，返回时一个字都还没生成。
  • 循环结束靠 finish_reason（stop=自然说完 / length=被 max_tokens 砍断）+ [DONE]。max_tokens 是服务器踩的刹车，砍断也照发终止信号，不会因回复过长而无限阻塞；真正卡死只有网络失联 → 靠 client(timeout=…) 兜底。
  • try/except 要包住 for 循环本身——数据在循环里懒加载，只包在 create() 上接不住流式中途的网络错误。
stream = client.chat.completions.create(
  model="deepseek-chat", messages=messages, stream=True,
  stream_options={"include_usage": True},   # ← 没这行，流式下 usage=None
)
for chunk in stream:
  if not chunk.choices:                 # 最后一个 chunk：choices 空，只带 usage
    if chunk.usage: total = chunk.usage.total_tokens
    continue
  piece = chunk.choices[0].delta.content   # 正文就在 delta.content
  if piece: print(piece, end="", flush=True)  # 逐字打印
口诀正文看 delta.content、usage 藏在空 choices 的尾 chunk、结束看 finish_reason——阻塞式 next 自动对齐，try 要包住整圈循环。
